# Chatbot: route every response through Claude Haiku

## Context

`server/services/chatbot.service.js` was migrated from Groq to Claude Haiku
(`claude-haiku-4-5`) in commit `79340673`. The catalog-grounded Q&A path and
image analysis already call Claude correctly. This change closes the
remaining gaps so that every customer-facing reply is generated by the model
rather than static or rule-based text, and fixes a bug that's currently
breaking full-catalog grounding.

## 1. Fix the catalog-grounding bug

`buildFullCatalogContext` (chatbot.service.js, in the per-product loop)
references an undefined variable `hasDiscount` when it means the
locally-computed `anyOnSale`. This throws a `ReferenceError` inside the
function's `try/catch` on every call that reaches a product with
`computedPrices.length > 0`, is silently swallowed, and the function returns
`null`. Effect: `handleChatbotQuery` never gets the full-catalog context
(`catalogContext = fullCatalog || productContext`), so Claude only ever sees
the narrower per-query `productContext` (top ~6 search results), not the
full shop.

Fix: replace `if (hasDiscount)` with `if (anyOnSale)`.

## 2. Greeting via Claude

Today `getGreetingResponse()` returns a non-async object built from
`getGreeting()`, which picks one of 4 hardcoded strings.

New `getGreetingResponse` becomes `async`:
- Calls `callClaude` with a short instruction prompt (e.g. "A new customer
  just opened the chat widget. Greet them warmly as DrinksHarbour AI in
  2-3 sentences, invite them to ask about drinks, prices, or event
  planning.") against `BASE_SYSTEM_PROMPT`.
- On success, `response` is the model's text.
- On failure (`callClaude` returns `null`), `response` falls back to a single
  fixed string: `"👋 Hi! I'm your DrinksHarbour assistant. Ask me about
  drinks, prices, or planning an event!"` — not the old 4-string rotation.
- `quickReplies` and `categories` stay exactly as they are today (static
  UI button data, not generated text — no reason to involve the model).

All three call sites (`handleChatbotQuery`'s empty-query short-circuit, the
`intent.isGreeting` short-circuit, and the catch-all error handler) must
`await` the now-async function. The catch-all error handler (top-level
`try/catch` in `handleChatbotQuery`) switches to a plain minimal error
string instead of a greeting — a DB/unexpected error isn't a fresh greeting
moment, and re-greeting on error could itself recurse into another failing
Claude call.

## 3. Consolidate the post-Claude fallback

Current flow in `handleChatbotQuery`:
1. Call Claude with full system prompt + catalog.
2. If null AND `conversationHistory.length > 0`, retry with a minimal prompt.
3. If still null: re-check for greeting, check for an exact product-name
   match (hand-formatted string), else list top 3 products
   (hand-formatted), else call `generateFallbackResponse` (static
   knowledge-base lookup or generic "not sure about that" text).

New flow:
1. Call Claude with full system prompt + catalog.
2. If null, retry once with a minimal prompt — unconditionally (not gated
   on conversation history; a fresh single-turn message deserves the same
   retry as a multi-turn one).
3. If still null, return one honest fixed string:
   `"I'm having trouble answering right now — please try again in a
   moment, or browse /shop."`
   Product result cards (the separate `products` array in the response,
   rendered by the widget as cards) are computed independently from DB
   search results and keep working regardless of whether Claude succeeded
   — only the `response` text changes.

This removes the need for `generateFallbackResponse`, `getKnowledgeSnippet`,
and the `beverageKnowledgeBase` object. Confirmed via repo-wide grep that
none of these three are referenced anywhere outside this file/this fallback
path, so they become dead code and are deleted (~115 lines).

## 4. File-upload (drink list) summary via Claude

Current `handleFileQuery` parses uploaded lines into `{quantity, name}`,
looks each up via `queryProducts` (deterministic DB matching — unchanged),
then builds the customer-facing summary with string concatenation
("📄 Processed N items...", per-line `• name xN = ₦total`, computed grand
total, free-delivery nudge, "Not in catalog" list), and makes a *second*,
separate Claude call only to suggest alternatives for not-found items.

New: keep all parsing/matching/price math in JS exactly as-is (it must stay
exact — these are real prices and quantities). Instead of concatenating the
final string, pass the computed data as context to a single Claude call:

```
ORDER ITEMS REQUESTED:
• <name> x<qty> = ₦<line total>   (for each matched item)
GRAND TOTAL: ₦<total>
ITEMS NOT FOUND IN CATALOG: <names, or "none">
FULL SHOP CATALOG: <existing buildFullCatalogContext() output, for alternatives>
```

System prompt instructs Claude to present the order clearly (bullets,
quantities, unit/line totals, grand total), mention the existing free
delivery threshold when relevant, and suggest catalog alternatives for any
not-found items — using `BASE_SYSTEM_PROMPT`'s pricing rules so it never
invents a price. This replaces both the old hand-built string and the old
separate "alternatives" call with one Claude call.

If Claude fails here, fall back to the literal computed summary in the old
hand-built format — this data is exact factual order math (not generated
prose pretending to be a smart answer), so showing it verbatim on AI failure
is honest and useful, unlike the rule-based text removed in section 3.

## Out of scope

- No streaming / SSE — the existing controller is a plain request/response;
  adding streaming would require client changes neither widget currently
  supports. Not requested.
- No change to `extractIntent` (regex-based intent/filter extraction used to
  query the DB) — it's a routing/search-filter step, not customer-facing
  text, so it stays.
- No change to `analyzeImage` / `handleImageQuery` text fallbacks (e.g. "I
  had trouble reading the image...") — these are already honest
  safety-net messages, not rule-based "smart" answers.
- `generateProductDetails` (exported from this file) is unused by any route
  or caller — left untouched, out of scope.

## Verification plan

1. Start the server locally (`ANTHROPIC_API_KEY` set).
2. `POST /api/chatbot/greeting` twice — confirm two different,
   model-generated greetings (not from the old static set).
3. `POST /api/chatbot/query` with a price-style query (e.g. "price of
   Hennessy") — confirm response includes on-sale tagging without the
   service throwing/logging the `hasDiscount` error, i.e. full catalog
   context is now reaching Claude.
4. Temporarily break `ANTHROPIC_API_KEY` (or point at an invalid model) and
   re-run the greeting + a query — confirm the new single honest fallback
   string appears instead of the old canned/rule-based text, and the
   process doesn't crash.
5. Upload a small drink-list file via the widget — confirm the order
   summary is Claude-composed (varies in phrasing) but the totals/quantities
   match the deterministic math exactly.
